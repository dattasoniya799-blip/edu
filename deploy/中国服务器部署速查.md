> 面向:阿里云等**国内** ECS(Ubuntu 22.04/24.04)。这份是「踩坑后的最短路径」,
> 配合一键脚本 `deploy/cloud-deploy.sh`,顺利的话 20~40 分钟(主要耗在下载依赖)。
> 首次实战记录:2026-07-04 部署到阿里云 121.41.70.11 成功(全绿)。

## 0. 一句话结论 / 最快路径

**能 SSH 就 SSH;SSH 被封就走 Workbench。** 代码放 GitHub(临时公开)→ 服务器自己 `git clone` →
跑 `deploy/cloud-deploy.sh` 一条命令搞定(装 Docker+配国内加速+构建+启动+建管理员+冒烟)。

```bash
# 在服务器上(SSH 或阿里云 Workbench 终端,root):
cd /opt && rm -rf qiming && git clone --depth 1 <仓库HTTPS地址> qiming && cd qiming && \
  LLM_API_KEY='sk-你的DeepSeekKey' ORG_NAME='机构名' ADMIN_NAME='管理员' ADMIN_PHONE='13900000001' \
  nohup bash deploy/cloud-deploy.sh > /tmp/deploy.log 2>&1 &
# 看进度:tail -40 /tmp/deploy.log   直到出现 "=== 部署完成 ==="
```

管理员密码在日志 `[5/6]` 段打印一次(不传 `ADMIN_PASSWORD` 时自动生成)。

## 2. 四个必踩的坑与解法(重要)

### 坑① 客户端 SSH 被阿里云边缘封禁(最耗时的坑)
- **现象**:本机 `ssh` 反复 `Connection closed by <ip> port 22`;**重启实例、关 fail2ban、清 iptables 都没用**;
  服务器上 `journalctl`/`iptables`/`nft` 里**查不到你的 IP**(说明包根本没到实例)。
- **根因**:短时间内多次快速连接触发了阿里云**边缘的暴力破解/扫描防护**,把你的**公网 IP 整个拉黑**
  (不分端口、跨重启、本机改不掉)。**是被"快速重连"触发的——不要连续重试!**
- **预防**:首次连服务器**一次成功**,失败了**别马上重试**;把多条命令合并进**一次 SSH 会话**。
- **解法**(任选,推荐 B):
  - A. 停手等它自动过期(时间不定,可能很久,不推荐干等);
  - B. **改用阿里云 Workbench(远程连接)**——走控制面、不受 IP 封锁影响,直接拿到 root 终端,
    然后按"最快路径"让服务器自己拉代码部署。**本次就是这么救回来的。**
  - C. 去「云安全中心」把你的 IP 解封/加白名单(位置不好找,不如 B 直接)。

### 坑② Ubuntu 24.04 的 SSH 是 socket 激活(想加端口时会踩)
- 24.04 用 `ssh.socket` 决定监听端口,`sshd_config` 里的 `Port` **被忽略**。
- 要加端口:改 `ssh.socket`(`systemctl edit ssh.socket` 加 `ListenStream=`),
  或直接 `systemctl disable --now ssh.socket && systemctl enable --now ssh`(回传统模式,`sshd_config` 的 `Port` 才生效)。
- 正常部署**用不到**这个;只有想临时借端口绕封锁时才会遇到。

### 坑③ 服务器连不上 Docker Hub(构建第一步就失败)
- **现象**:`failed to resolve ... registry-1.docker.io ... i/o timeout`。
- **解法**:配国内镜像加速(**`cloud-deploy.sh` 现已自动做**;手动如下):
  ```bash
  mkdir -p /etc/docker && printf '%s\n' '{"registry-mirrors":["https://docker.m.daocloud.io","https://docker.1ms.run","https://hub-mirror.c.163.com","https://mirror.baidubce.com"]}' > /etc/docker/daemon.json && systemctl restart docker
  # 验证:docker pull docker/dockerfile:1  应能拉到
  ```
- 公共镜像有时会失效;最稳是**阿里云自己的加速器**(容器镜像服务 ACR → 镜像加速器,复制专属 `https://xxxx.mirror.aliyuncs.com` 填进上面数组)。

### 坑④ 构建时装 python3/make/g++ 很慢(不是卡死)
- 后端镜像要装编译工具(argon2 等原生模块需要),`apt` 从**海外 Debian 源**拉,国内可能慢到 30~40 分钟。
- **在下载就不是卡死**(日志里包一个个 `Get:` 在下),耐心等即可;要提速可给后端 Dockerfile 的 apt 换国内 Debian 源(可选,未默认改)。

## 3. 私有仓库怎么送上服务器(SSH 不通时)

服务器拉不了私有库。本仓库 **已确认无任何密钥入库**(`.env*` 全 gitignore,`.env.development` 只有 `VITE_USE_MOCK=true`)。
所以最简单:**GitHub → Settings → Danger Zone → 临时 Change to public → 服务器 `git clone` → 立刻改回 private**。
(更安全的替代:建只读 Deploy Key / 细粒度 PAT,用带 token 的 clone URL,保持私有。)

机密项(DeepSeek Key 等)**不进仓库**,只在 `cloud-deploy.sh` 的**运行命令里用环境变量传入**,脚本会写进服务器本地 `deploy/.env.prod`(gitignore、chmod 600)。

## 坑⑤ 非标准端口(8081/8082)在国内网络被拦 → 三端收到 80 端口下(2026-07-04 实战)
- **现象**:学生端(80)能开,管理端/教师端(8081/8082)浏览器报 **502**;但服务器本地 `curl localhost:8081`、
  甚至 `curl <公网IP>:8081` 都是 **200**。说明服务器完全正常,是**浏览器到阿里云之间的网络路径**在非标准端口上返回 502
  (国内家庭/学校/单位网络常只放行 80/443)。
- **解法(已改为默认架构)**:三端同托管在 **80 端口**,用路径区分:`/`(学生)、`/admin/`(管理)、`/teacher/`(教师)。
  实现 = 管理端/教师端构建时 `VITE_BASE=/admin/`(见 web.Dockerfile)+ 路由 `basename` 取 `import.meta.env.BASE_URL`
  + nginx 单 server 按路径 `try_files` 到各自 dist。前端 API 仍走绝对 `/api/v1`、socket.io 走同源 `/socket.io`,不受影响。
- 有域名后可回退三子域(各 listen 80 + server_name,base 回 '/'),HTTPS 全在 443 也无此问题。
